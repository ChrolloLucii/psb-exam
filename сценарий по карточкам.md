# Markdown — лекция по карточкам

---

## ЧАСТЬ 0. Завязка

### Карточка 1 — Титул
**Суть.** Стартовый слайд: одно слово — `Markdown`.

  «Сегодня про штуку, на которой держится половина интернета, а узнаёт о ней нормально хорошо если каждый десятый.»

---

### Карточка 2 — Серая стена vs оформленный текст
**Суть.** Два варианта одного и того же текста рядом: слева сплошняк, справа оформленный.

  «Слева — стена текста. Справа — то же самое, но с заголовком, списком, выделениями. Какой приятнее читать? Ответ очевиден. Вот как делать правую версию — и есть тема урока. И на это уйдёт не пара часов, а пара минут.»

**🧠** это крючок на весь урок — к нему можно вернуться в конце.

---

## ЧАСТЬ 1. Что это и зачем

### Карточка 3 — Что такое Markdown
**Суть.** Язык разметки: пишешь в любом редакторе, открывается везде, форматирование видно прямо в тексте.

  «Markdown — это язык разметки. Пишешь хоть в nano, хоть в VS Code — откроется где угодно и не сломается. Главная фишка: форматирование задаётся символами прямо в тексте, никаких невидимых настроек под капотом, как в Word. Поэтому тут не бывает “таблица съехала, потому что я что-то нажал”.»

**🧠** Word **прячет** форматирование, Markdown — **показывает**.

---

### Карточка 4 — Почему все на нём сидят
**Суть.** Простой, стабильный, конвертируется куда угодно. Принцип в три шага.

  «Вся серьёзная документация, почти весь опенсорс, нормальные блоги — на Markdown. Почему? Потому что он просто работает. Один и тот же текст легко превращается в сайт, в PDF, в презентацию — без визуального редактора и менюшек на пол-экрана.»

**Принцип в три шага (на слайд):**
```
печатаешь символы → программа их читает → получаешь красоту
```

**🧠** Windows vs Linux. В винде, чтоб что-то сделать, надо продираться сквозь меняющийся UI; в линуксе открыл терминал и по инструкции 10-летней давности сделал всё за пару команд. Markdown — это «терминал» для текста.

## ЧАСТЬ 2. История и где живёт

### Карточка 5 — Откуда взялся
**Суть.** 2004, Джон Грубер и Аарон Шварц. Идея: черновик = готовый результат.

  «Придумали в 2004-м Джон Грубер и Аарон Шварц — тот самый, что приложил руку к Reddit и RSS. Они не стали городить монстра с тоннами тегов, наоборот — сделали максимально просто. Идея почти гениальная в своей лени: писать так, чтобы исходник было приятно читать и без рендера. Открыл `.md` хоть в блокноте — всё понятно.»

**🧠** на GitHub первое, что видишь в любом проекте — `README.md`. Это «лицо» репозитория. И оно на Markdown не просто так.

---

### Карточка 6 — Где ты его уже видел
**Суть.** GitHub, Discord, Reddit, Telegram, Notion, Obsidian.

 «Главный поинт: ты уже писал на Markdown и не знал. Выделял в Discord текст звёздочками `**вот так**`? Это он. Оформлял сообщение в Telegram? Снова он. Он работает настолько незаметно, что его не замечаешь.»

**Почему именно он, а не своя разметка у каждого:** сервису проще взять готовый понятный стандарт, чем изобретать велосипед. Markdown стал этим стандартом — потому что простой и открытый.

**🧠** хороший инструмент тихо встраивают везде. Поэтому Markdown повсюду.

## ЧАСТЬ 3. Базовый синтаксис

> Каждый элемент показываем в формате **«ты пишешь» → «видишь»**. Слева сырой текст, справа результат.

### Карточка 7 — Главный принцип: символ-обёртка
**Суть.** Якорная мысль всего блока.

  «Во всём Markdown работает один принцип — символ-обёртка. Ты оборачиваешь текст определёнными символами, и он меняет вид. Запомнишь это — считай, поняла половину.»

---

### Карточка 8 — Заголовки
**Пишешь:**
```
# Большой заголовок
## Поменьше
### Ещё меньше
```

 «Решётка `#` — это заголовок. Одна — самый большой, как название главы. Две — подзаголовок. Три — ещё мельче. Чем больше решёток, тем мельче текст.»

**🧠** **пробел после решётки обязателен!** Без пробела не сработает — это ошибка №1 у новичков.

---

### Карточка 9 — Жирный, курсив, зачёркнутый
**Пишешь → видишь:**
```
**жирный**           → жирный
*курсив*             → курсив
~~зачёркнутый~~      → зачёркнутый
***жирный курсив***  → и то и другое сразу
```
 «Жирный — **две** звёздочки с каждой стороны. Курсив — **одна**. Зачёркнутый — две тильды `~~`.»

**🧠** зачёркнутый удобен для шуток и анонсов: `Стрим ~~завтра~~ **сегодня** в 19:00`. И не спамить выделениями — если жирное всё, то жирное ничто.

---

### Карточка 10 — Списки
**Пишешь:**
```
Маркированный:        Нумерованный:
- молоко              1. Рождение
- хлеб                2. установка arch
- кот                 3. арч ломается
```

 «Маркированный — **дефис и пробел** в начале строки. Нумерованный — **цифра, точка, пробел**.»

**🧠** прикол с нумерованным — можешь хоть везде писать `1.`, Markdown сам посчитает по порядку.

---

### Карточка 11 — Ссылки и картинки
**Пишешь:**
```
[текст ссылки](https://youtube.com)
![подпись к картинке](cat.png)
```

  «Ссылка: в **квадратных** скобках — текст, который видно, в **круглых** — куда ведёт. Картинка — то же самое, но впереди ставишь **восклицательный знак** `!`.»




---

## ЧАСТЬ 4. Код, таблицы, цитаты

### Карточка 12 — Код
**Два вида.**

Инлайн (внутри строки) — одинарные кавычки:
```
Нажми `Ctrl + C`, чтобы скопировать.
```

Блок (многострочный) — три обратные кавычки:
````
```
print("I use arch btw!")
```
````

  «Одинарными кавычками оборачиваешь код внутри предложения — название клавиши, команду, имя файла. Тремя — целый блок кода, он встанет в рамочку моноширинным шрифтом.»

**🧠  ** одна кавычка — словечко, три кавычки — кусок кода в рамке.

---

### Карточка 13 — Таблицы
**Пишешь:**
```
| Игра     | Оценка |
|----------|--------|
| заглушка | 10/10  |
| заглушка | 8/10   |
```

  «Колонки разделяешь палочкой `|`. Вторая строка из дефисов — это “шапка”, она отделяет заголовки от данных. Палочки можно не ровнять идеально — Markdown сам выровняет.»

**🧠** главное — строка из дефисов под заголовками. Нет её — таблицы нет.

---

### Карточка 14 — Цитаты
**Пишешь:**
```
> Это цитата.
```
> Интеллект — это способность избежать выполнения работы, но тем не менее сделать так, чтобы она была выполнена. © Линус Торвальдс

  «Знак “больше” `>` в начале строки делает цитату — текст с полоской слева. Удобно для важных мыслей, правил, чужих слов. На форумах и в Telegram так оформляют ответ на сообщение.»

---

## ЧАСТЬ 5. Кульминация — собираем настоящий README

### Карточка 15 — Всё вместе
**Суть.** Показываем, как все элементы складываются в реальный `README.md`, как у проектов на GitHub.

**Готовый пример на слайд:**
````
# TeleBot 🤖

Простой телеграм-бот для напоминаний.

## Возможности
- напоминания по времени
- список дел
- работает 24/7

## Установка
```
git clone https://github.com/me/telebot
pip install -r requirements.txt
python bot.py
```

## Команды
| Команда   | Что делает          |
|-----------|---------------------|
| /start    | запускает бота      |
| /remind   | ставит напоминание  |

> ⚠️ Нужен Python 3.10+
````

  «Вот так выглядит “прод” — реальный документ из любого проекта. Заголовок, описание, список фич, установка блоком кода, таблица команд. Всё, о чём мы говорили — здесь. Вы только что научились читать и писать то, на чём держится вся документация и опенсорс.»

> 💡 Тут можно открыть реальный README популярного проекта на GitHub.

## ЧАСТЬ 6. За пределами базы

### Карточка 16 — Расширения
**Суть.** Голый Markdown — это база. Дальше — надстройки, не ломающие принцип «пишешь символами → получаешь результат».

**Говоришь коротко про три штуки:**

- **GFM (GitHub Flavored Markdown)** — «Markdown как все привыкли». Добавил таблицы, чек-листы `- [ ]` / `- [x]`, подсветку кода, эмодзи `:fire:`, автоссылки.
- **Mermaid** — пишешь текстом описание схемы → получаешь нарисованную диаграмму. Не таскаешь стрелочки мышкой:
  ````
  ```mermaid
  graph LR
      A[6] --> B[7]
      B --> C[Газан]
  ```
  ````
  Поменялась логика — поправил строчку, схема перерисовалась сама.
- **LaTeX-формулы, сноски и прочее** — под любую задачу есть надстройка.

**🧠** всё это по-прежнему обычный текстовый файл. Открыл в блокноте — читается. Расширения не ломают простоту, они её докручивают.

---

### Карточка 17 — MkDocs: из папки с `.md` — в сайт
**Суть.** Из Markdown можно собрать не один документ, а целый сайт-документацию.

  «Кидаешь свои `.md` в папку, пишешь маленький конфиг `mkdocs.yml`, команда `mkdocs serve` — и локально крутится готовый сайт с навигацией, поиском и тёмной темой. Правишь текст — страница сама перезагружается. `mkdocs build` + деплой — и он в интернете. Ни строчки HTML или CSS.»

**🧠** тот же принцип на максималках — контент отдельно (твой текст), оформление отдельно (тема). Никаких CMS-комбайнов и «шаблон поехал».

> 💡 Крутой пример (официальная дока Material for MkDocs): https://squidfunk.github.io/mkdocs-material/

---

### Карточка 18 — Где писать
**Суть.** Инструменты, чтобы попробовать прямо сейчас, ничего не ставя.

- **dillinger.io** — слева пишешь, справа сразу результат.
- **stackedit.io** — то же, чуть больше возможностей.

# 🧪 ПРАКТИКА (на выбор)

> Все материалы ниже готовы к выдаче — можно прямо кинуть на слайд или в чат.
> Порядок по сложности: 1 — разминка, 4 — кульминация.

## Практика 1 — «Причеши стену» (разминка, ~5 мин)
*После карточки 11 — когда прошли заголовки, выделения, списки, ссылки.*

Дай студентам плоский текст и попроси оформить: заголовок, название игры жирным, список фич, ссылку.

**Выдать на слайд:**
```
портал 2 это головоломка от valve главный герой берет портальную
пушку и проходит уровни фишки кооператив физика юмор играть тут
https://store.steampowered.com
```

**Задача:** заголовок «Portal 2», название игры жирным, 3 фичи списком, слово «играть тут» сделать ссылкой.

---

## Практика 2 — «Карточка-обзор в таблице» (~5–7 мин)
*После карточки 13.*

**Задача:** сделать таблицу из 3 строк: что-то любимое (игра / фильм / книга), оценка, одна строка мнения.

**Пример-образец:**
```
| Что        | Оценка | Мнение                  |
|------------|--------|-------------------------|
| Portal 2   | 10/10  | идеальный кооп          |
```

---

## Практика 3 — «Найди баг» (закрепление, ~3–5 мин)
*Отлично заходит после базового синтаксиса — ловит типичные ошибки.*

**Выдать сломанный код, попросить починить:**
```
#Мой проект
Это **очень крутой проект.

-первый пункт
-второй пункт

| Имя | Оценка
|-----|
| бот | 10/10 |
```

**Спрятано 4 ошибки** (ответ для тебя):
1. Нет пробела после `#` → не станет заголовком.
2. Незакрытый `**` → жирное «убежит» дальше по тексту.
3. Нет пробела после `-` → список не соберётся.
4. В таблице кривая шапка: число столбцов в заголовке и разделителе не совпадает.

---

## Практика 4 — «README для бота» (кульминация, ~10–15 мин)
*После карточки 15 — собирает вообще всё.*

**Задача:** написать настоящий `README.md` для выдуманного бота. Обязательные элементы:
- заголовок проекта (`#`) + эмодзи;
- краткое описание одной строкой;
- список возможностей (`-`);
- блок установки (```` ``` ````) с парой команд;
- таблица команд;
- цитата-дисклеймер (`>`).

Можно дать заготовку «рыбу» из карточки 15 и попросить переделать под свой проект.

---

## Практика 5 — Бонус для шустрых
*Кто справился раньше всех.*

- **Чек-лист:** сделать список дел с галочками `- [ ]` и `- [x]`.
- **Mermaid:** нарисовать схему из 3 блоков (например `Идея --> Код --> Прод`).
- Открыть https://dillinger.io и проверить там же.

# 📋 Шпаргалка (раздать / финальный слайд)

| Хочешь        | Пишешь                         |
|---------------|--------------------------------|
| Заголовок     | `# Текст` (пробел обязателен!)  |
| Жирный        | `**текст**`                    |
| Курсив        | `*текст*`                      |
| Зачёркнутый   | `~~текст~~`                    |
| Список        | `- пункт`                      |
| Нумерованный  | `1. пункт`                     |
| Ссылка        | `[текст](url)`                 |
| Картинка      | `![подпись](файл.png)`         |
| Код в строке  | `` `код` ``                    |
| Блок кода     | ```` ``` ```` … ```` ``` ````   |
| Таблица       | `\| A \| B \|` + строка `\|---\|---\|` |
| Цитата        | `> текст`                      |

> Один принцип на всё: **оборачиваешь текст символами — он меняет вид.**
